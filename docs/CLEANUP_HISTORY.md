## Cleanup Criteria

### Files Safe to Delete

1. **Completed Feature Summaries**: Documentation created during feature development that describes implementation details, mockups, comparisons, or "before/after" analysis for features that are already shipped and merged.
   - Example patterns: `*_SUMMARY.md`, `*_COMPARISON.md`, `*_MOCKUPS.md`, `*_WIREFRAMES.md`
   - Rationale: The actual implementation is in the codebase; summaries are redundant once merged

2. **Auto-Generated Files**: Files that are created by scripts and can be regenerated at any time.
   - Example: Files generated by `scripts/roadmap-tracker.ts`
   - Rationale: Can be recreated on-demand, shouldn't be in version control

3. **Temporary/Scratch Files**: Files created for temporary purposes during development.
   - Example patterns: `*.tmp`, `*.temp`, `*.bak`, `*~`, `.DS_Store`
   - Rationale: Not needed for production or collaboration

4. **Duplicate Documentation**: Content that exists in multiple places.
   - Example: Multiple summary files for the same feature
   - Rationale: Keep canonical version, delete duplicates

### Files to Keep in Root Directory

Only high-level, essential project documentation should remain in the root:

1. `README.md` - Main project entry point
2. `FEATURES.md` - Feature catalog
3. `ROADMAP.md` - Future planning
4. `CHANGELOG.md` - Version history
5. `CONTRIBUTING.md` - Contribution guidelines
6. `SECURITY.md` - Security policy
7. `MANUAL_TESTING.md` - Testing procedures
8. `LICENSE` - License file

### Where to Put Implementation Documentation

- **Feature implementations** → `docs/features/`
- **Architecture designs** → `docs/architecture/`
- **Requirements/FRDs** → `docs/requirements/`
- **Setup guides** → `docs/setup/`
- **User guides** → `docs/user-manual.md` or similar

## Future Maintenance

### Guidelines for Adding New Documentation

1. **Implementation docs** should go in `docs/features/` or `docs/architecture/`
2. **Only summaries** should be in root-level `FEATURES.md` or `ROADMAP.md`
3. **Update indexes** when adding new documentation (`docs/README.md`, `docs/features/README.md`)
4. **Delete summaries** after features are shipped (implementation is the documentation)

### When to Clean Up

Perform cleanup when:
- A major feature is completed and merged
- The root directory has more than 10 markdown files
- Auto-generated files accumulate (check `.gitignore`)
- New developers report difficulty finding documentation
